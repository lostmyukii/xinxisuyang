# AGENTS.md

本文件适用于仓库根目录及全部子目录。任何自动化 Agent、开发者或后续维护者开始工作前，必须先阅读本文件和根目录下的 `开发文档.md`。

## 1. 项目事实

- 项目是“信息素养大赛成绩核对系统”。
- 唯一外部数据入口是 `https://www.kdocs.cn/wo/sl/v130lGtJ`。
- 比赛管理电脑是当前这台 macOS 电脑。
- 比赛时使用 Google Chrome，管理员本人先登录金山文档并保持目标页面打开。
- 允许安装 Chrome 扩展和本机采集服务。
- 当前不依赖金山开放平台 API、APPID、APPKEY 或 OAuth。
- 金山文档是裁判录入源；本系统只能读取，不得修改。

## 2. 工作前的来源层级

遇到冲突时，按以下顺序判断：

1. 当前已登录 Chrome 中目标金山文档的实际结构与总记录数。
2. 本机最近一次通过完整性校验的成功快照。
3. `开发文档.md` 中已确认的业务规则和安全边界。
4. 测试夹具和开发模拟数据。

不得用截图、缓存页面、旧导出文件或测试数据覆盖更高层级的事实。

## 3. 不可突破的边界

- 不向金山文档写入、编辑、排序、筛选、移动或删除任何内容。
- 不要求用户把金山账号密码、Cookie、短信验证码或浏览器配置文件交给项目。
- 不绕过金山登录、分享权限、访问控制或验证码。
- 不将原始成绩、手机号、身份证号或完整快照上传第三方服务。
- 不把 OCR、截图识别或 Canvas 像素抓取作为正式采集方案。
- 不在无法证明完整读取时宣称“自动同步成功”或“实时数据”。
- 不把附近文件夹、其他表格或模拟数据当成当前赛事的正式来源。
- 不默认开放局域网或公网端口；本机服务默认只绑定 `127.0.0.1`。

## 4. 已确认业务规则

### 4.1 排名分区

固定按以下组合分别排名：

```text
赛区 × 赛项 × 组别
```

三个维度中的任何一个不同都不得混排。

### 4.2 同分规则

使用标准竞赛排名：同分同名次，下一名跳号。

```text
1、2、2、4
```

### 4.3 成绩规则

- 每个赛项分别配置最低分和最高分。
- 不提供 `0–100` 默认范围。
- 最多允许两位小数。
- 空值、非数字、越界、超过两位小数的记录不参与排名。
- 未配置成绩范围的赛项标记“待配置”，不生成该赛项排名。
- 系统不得自动修正裁判填写的成绩。

### 4.4 两套视图

- 管理后台：同步、完整核对、异常、配置、快照和导出。
- 展示大屏：仅显示赛区、赛项、组别、脱敏姓名、成绩、名次和更新时间。
- 展示大屏不得接收完整敏感记录后再用 CSS 隐藏；后端必须生成独立脱敏 DTO。

## 5. 自动采集的通过门

自动采集必须先通过技术探针。至少验证：

- 连续 3 次重载字段与记录数一致。
- 记录数与金山页面显示总数一致。
- 水平或垂直滚动不影响完整性。
- 必需字段全部存在。
- 能稳定去重并识别修改。
- 一个同步周期内能发现一条成绩变化。
- 采集过程没有任何写操作。

如果任一条件失败：

1. 将自动采集标记为“验证未通过”。
2. 禁止候选数据成为已发布快照。
3. 保留手动复制或文件导入能力。
4. 在交付说明中明确真实边界。

## 6. 目标架构与模块边界

```text
apps/extension  ->  apps/server  ->  packages/domain + packages/ranking
                                      |
                                      v
                                packages/storage
                                      |
                                      v
                                   apps/web
```

- `apps/extension`：只处理目标金山页面的连接和只读采集。
- `apps/server`：同步调度、运行时校验、本地 API、导出编排。
- `apps/web`：管理后台和展示大屏，不直接访问金山文档。
- `packages/domain`：规范字段、Schema、异常码、公共 DTO。
- `packages/ranking`：无 IO 的纯排名逻辑。
- `packages/storage`：SQLite、事务、快照、加密列。
- `packages/ui`：iOS 风格设计令牌和共享组件。

模块之间通过明确类型与 Schema 通信。不要把采集、排名、数据库和页面逻辑堆进一个文件。

## 7. Chrome 扩展规则

- 使用 Manifest V3。
- Host permission 只覆盖目标金山域名与 `127.0.0.1` 本机接口。
- 页面上下文桥接与扩展隔离环境之间必须校验消息来源和结构。
- 扩展不得持久化 Cookie、账号令牌和无关页面内容。
- 扩展不得访问其他标签页，除非目标页识别流程确实需要且已在权限说明中写明。
- 采集结果发送前先做字段白名单和大小限制。
- 连接本机服务时使用安装生成的随机配对令牌。
- 任何金山网页适配逻辑必须集中封装，并有结构变化测试；不得散落在 UI 组件中。

## 8. 同步与发布不变量

- 自动同步默认 60 秒一次；立即同步与其共用同一队列。
- 同一时刻只运行一个同步任务。
- 每次同步先形成候选快照，完整校验后才能发布。
- SQLite 写入和当前快照切换必须处于同一事务。
- 失败任务不得覆盖最近成功快照。
- 内容哈希不变时不得生成重复快照。
- 120 秒未成功同步显示“数据可能已过期”；5 分钟显示严重警告。
- 大屏必须始终显示最后成功时间和数据状态。

## 9. 排名实现不变量

- 使用十进制精确类型或定点整数，禁止直接用 JavaScript `number` 比较成绩。
- 排名函数必须是纯函数，输入记录和赛项规则，输出排名与异常列表。
- 分区键必须包含赛区、赛项和组别。
- 同分排名使用 `1、2、2、4`。
- 同分展示顺序稳定，但不得用来源顺序拆分名次。
- 无效成绩必须保留原始文本和明确异常码，但不进入有效排名数组。

## 10. 隐私与测试数据

- 手机号和身份证号按列加密，密钥进入 macOS Keychain。
- 日志只记录记录 ID、字段名、计数、错误码和哈希，不记录真实敏感值。
- 测试、截图、Storybook、演示和文档只能使用虚构或脱敏数据。
- 禁止把当前金山页面的真实姓名、手机号、身份证号复制进仓库。
- `data/`、导出文件、SQLite 数据库、浏览器抓包和调试快照必须被 Git 忽略。
- 若需要提交失败样本，先生成结构等价的合成样本。

## 11. iOS 视觉契约

用户已确认 A“赛事指挥台”的 iOS 风格方向。

### 11.1 固定令牌

- Frost：`#F5F7FA`
- Ink：`#101114`
- System Blue：`#0A84FF`
- Action Blue：`#0066CC`（浅色背景的小字号链接、按钮和焦点）
- System Green：`#30D158`
- System Orange：`#FF9F0A`
- System Red：`#FF453A`

### 11.2 字体

- 标题：SF Pro Display / PingFang SC / 系统无衬线。
- 正文：SF Pro Text / PingFang SC / 系统无衬线。
- 成绩、名次、时间：SF Mono / 等宽回退字体，并启用表格数字。

### 11.3 视觉重点

- 后台使用浅色磨砂、大标题、克制圆角和分段筛选。
- 大屏使用深色高对比和大字号排名。
- 项目唯一重点识别元素是“数据新鲜度光环”。
- 状态光环同时包含文字和图标，不能只靠颜色。
- 不要把页面做成通用 KPI 卡片堆叠或 iPhone 设置页仿品。
- 首屏优先级固定为：同步可信度、异常处理、最新排名。

### 11.4 动效与可访问性

- 只在同步完成时使用一次短促、有意义的状态收束和差异高亮。
- 支持 `prefers-reduced-motion`。
- 保留清晰键盘焦点，达到 WCAG AA 对比度。
- `#0A84FF` 保留作深色背景或大面积品牌强调；浅色背景上的交互文字不得牺牲 AA 对比度。
- 错误文案说明问题和下一步，不使用模糊失败提示。
- 16:9 大屏与 Mac 常用桌面分辨率是首要适配目标，同时保证窄屏单列可用。

## 12. 导出规则

- 支持当前分组、当前赛项和全部赛项三种导出模式。
- 当前赛项文件按“赛区＋组别”拆分工作表。
- 默认字段：名次、赛区、赛项、组别、选手姓名、成绩、状态。
- 手机号和身份证号默认不导出；主动包含时必须二次确认并记录审计信息。
- 文件必须写入生成时间、快照 ID、排名规则、有效数、异常数和数据来源。
- Excel 生成失败时不得留下看似完整的残缺文件。

## 13. 代码与测试标准

- TypeScript 开启严格模式，禁止用大面积 `any` 绕过类型。
- 所有外部输入都通过运行时 Schema 校验。
- 排名、脱敏、字段映射和异常分类优先写成小型纯函数。
- 错误使用稳定错误码，UI 文案在展示层映射。
- 避免超大文件；一个模块只能有一个清晰职责。
- CSS 使用低特异性类和设计令牌，避免标签选择器覆盖组件状态。
- 每个修复至少补充能复现问题的测试。

项目脚手架建立后，根目录必须提供以下统一命令：

```bash
pnpm install
pnpm dev
pnpm lint
pnpm test
pnpm test:e2e
pnpm build
```

在脚手架尚未创建前，不得声称这些命令已经可运行。

## 14. 验证顺序

每次变更按风险从小到大验证：

1. 目标包的类型检查和单元测试。
2. 排名与异常规则测试。
3. 同步、SQLite 事务和快照集成测试。
4. 管理后台与大屏端到端测试。
5. Chrome 真实登录态下的只读采集验证。
6. 501 条当前规模和 5,000 条合成规模测试。
7. 4 小时现场稳定性测试。

真实金山页面验证不得修改原表。需要测试分数变化时，使用开发模拟页或由用户明确安排的非正式测试记录。

## 15. 工作纪律

- 保留用户现有文件和无关改动。
- 使用小步、可回退的修改。
- 不进行与当前目标无关的重构。
- 任何方案变化先更新 `开发文档.md`，再修改实现。
- 自动采集是否可用必须依据真实验证，不依据 UI 状态或模拟测试推断。
- 完成工作时报告：修改文件、验证命令、验证结果、仍存在的风险和人工步骤。
- 不得把“文档完成”“模拟数据通过”表述为“真实金山闭环已完成”。
